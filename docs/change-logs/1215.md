## Key decisions

- **CHECK constraint enforces kind/payload coupling** — bad data is rejected at the DB layer, not just the API.
- **Partial unique index** on `is_default WHERE is_default = 1` — at most one default at a time. `set_default_account` clears all in the same transaction before setting.
- **Secrets never in DB or HTTP response** — only opaque refs (`keychain:<service>` / `op://<vault>/<item>/<field>`). Plaintext is resolved at exec time only.
- No new top-level deps.

## Gotchas

- CHECK requires `config_dir IS NULL` (not empty string) on the api_key path and vice versa. `create_account` normalizes `""` → `None` at write time so callers can pass either.
- `update_account` rejects `id`, `kind`, `is_default`, `created_at`, `last_used_at`. Use `set_default_account` for the default flip.
- Phase 4 must add a **profile-level lock** (not per-task-slug like the existing executor lock) so two scheduler tasks bound to the same `config_dir` can't race the CLI's auth-refresh write.
