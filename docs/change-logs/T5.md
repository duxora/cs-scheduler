# T5 — Import existing Claude config-dir profiles as accounts

**Lets you register `~/.claude`, `~/.claude-fleet`, or any `~/.claude-profiles/*` as a scheduler account without re-running `claude /login`.**

## What changed

- `apps/scheduler/routes.py` — new `_has_claude_credentials(Path)` helper that probes the macOS Keychain (`security find-generic-password -s "Claude Code-credentials[-<sha256(abs)[:8]>]"`) and falls back to a `<dir>/.credentials.json` file check on Linux. Used by:
  - `/api/accounts/check` (fixed: previously always returned `has_credentials=false` on macOS).
  - new `GET /api/accounts/discover` — scans `~/.claude`, `~/.claude-fleet`, and every direct child of `~/.claude-profiles/`. Returns each candidate with `dir_exists`, `has_credentials`, `has_history`, `already_registered`, `registered_account_id`, and a derived `name_suggestion`.
  - new `POST /api/accounts/import` — registers an existing dir as `kind=config_dir` after validating dir + credentials + duplicate-config_dir. `skip_credentials_check=true` is the escape hatch for non-mac credential stores.
  - `AccountImport` pydantic model alongside `AccountCreate`.
- `src/claude_scheduler/cli.py` — `cs accounts discover` (table view) and `cs accounts import <name> --config-dir <path> [--plan-tier ...] [--default] [--no-credentials-check]`. Credential-probe logic duplicated locally to avoid an awkward cross-package import.
- `frontend/src/apps/scheduler/types/index.ts` — `AccountDiscoverCandidate`, `AccountDiscoverResponse`, `AccountImportPayload`.
- `frontend/src/apps/scheduler/lib/api.ts` — `discoverAccounts()` and `importAccount(payload)`.
- `frontend/src/apps/scheduler/pages/AccountsPage.tsx` — new "Import existing" button next to "+ New account". Picker modal lists candidates with status badges (creds / history / already-registered); rows that aren't importable are dimmed and unclickable. Step 2 collects name + plan tier, posts to `/import`, and refreshes the list.
- New tests:
  - `tests/test_routes_accounts_discover.py` (4 cases)
  - `tests/test_routes_accounts_import.py` (5 cases)
  - `tests/test_cli_accounts_import.py` (3 cases)

## Key decisions

- **Keychain over file probe.** The original `<dir>/.credentials.json` check was Linux-only; on macOS the Claude CLI stores OAuth tokens in the Login Keychain under a service name keyed by `sha256(abs_path)[:8]` (the default `~/.claude` path uses the unsuffixed `Claude Code-credentials`). Reverse-engineered by reading two existing keychain entries and matching them against `sha256(path)[:8]`. The probe runs both checks — keychain on macOS, file on any platform — so it Just Works on both.
- **Discover is server-side, not browser-side.** The browser can't read the Keychain or the user's home dir, so `/discover` runs the scan inside the FastAPI process and returns a flat candidate list. The FE doesn't need any local-fs access.
- **Picker UX over auto-import.** Each candidate is shown with its current state (creds present, history present, already registered) so the user can see why a row is or isn't importable. Non-importable rows stay visible but dimmed and disabled rather than being filtered out — the visibility itself is informative ("oh, I already imported that one").
- **Cross-app credential-probe duplication.** The CLI and the routes both need `_has_claude_credentials`. Instead of refactoring it into `claude_scheduler/core/`, I kept one copy in each module — this is two ~20-line functions and they share zero dependencies. Worth revisiting only if the probe gains real complexity (e.g. Linux secret-service support).
- **`skip_credentials_check` escape hatch.** On Linux without a `.credentials.json` (or on macOS if the keychain probe times out), the user can still register a dir by passing the flag. The FE auto-sets it when `has_credentials=false` because the user has already seen the badge and made a deliberate choice.

## Gotchas

- Both new GET/POST routes had to be registered *before* `@router.get("/api/accounts/{account_id}")` to avoid path-param shadowing — Codex put them in the right place but worth checking on any future addition.
- macOS uvicorn does not hot-reload (see `automation_hub_server_restart.md` in memory). Restart with the standard `kill PID && nohup uvicorn ...` recipe before clicking around.
- The CLI cannot import `_has_claude_credentials` from `apps.scheduler.routes` cleanly because that module imports FastAPI; spinning up a request stack inside a CLI invocation is overkill. Hence the duplication.
