## Key decisions

- **Build env from `os.environ.copy()`, never mutate `os.environ`.** Plaintext API keys live only in the env dict passed to `subprocess.run` — they go out of scope when the call returns.
- **Profile-level lock keyed by `sha1(config_dir)[:16]`** under `~/.claude-scheduler/profile-locks/`. Two concurrent runs against the same profile dir would race the CLI auth-refresh write; api_key accounts have no shared on-disk state so they skip the lock.
- **Resolution by name, not UUID** — `.task` files reference accounts by the human name (`# account: personal-pro`) so they survive renames in the DB if the CRUD API ever exposes a rename.
- **Failure modes:** unknown account name → `status=failed`; secret resolution error → `status=failed` with the resolver error surfaced in `error_message`; profile busy → `status=skipped` (matches the existing per-task-slug lock behavior).

## Gotchas

- The parser passes through unknown header fields as raw strings, so `# account: <name>` "just works" — the new field is absorbed by the existing kwargs sweep. Tests cover the path explicitly.
- `account_id` is only persisted on the run row when the task has an account configured. Tasks without an account leave the column NULL — the dashboard treats NULL/empty as "default account".
