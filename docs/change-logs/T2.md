# T2 — Accounts row enrichments + test button

**Second of three settings-UI tickets, builds on T1.**

## What changed

- `apps/scheduler/routes.py` — `_account_to_dict` now takes an optional `db` and, when provided, enriches each row with `runs_24h`, `failures_24h`, `cost_30d_usd`, `auth_failure_recent`, `health`. List + get handlers pass `db`; mutation handlers don't (no aggregation cost). New `POST /api/accounts/<id>/test` runs `claude -p "say ok" --max-turns 1` for 30s under the account's env and returns `{ok, exit_code, stderr_tail, took_ms}`. Always HTTP 200 (failures encoded in `ok:false`) so the FE doesn't need 4xx branching.
- `frontend/src/apps/scheduler/lib/api.ts` — `schedulerApi.testAccount(id)`.
- `frontend/src/apps/scheduler/lib/relativeTime.ts` (new) — "2h ago" / "3d ago" formatter.
- `frontend/src/apps/scheduler/types/index.ts` — `AccountHealth`, `AccountTestResult`; new fields on `Account` (all optional).
- `frontend/src/apps/scheduler/pages/AccountsPage.tsx` — row redesign with health pill (🟢 Active / 🟡 Idle / 🔴 Re-login / ⚪ Never used), runs/failures/cost meta line, "Copy re-login" button (config_dir only) writes `CLAUDE_CONFIG_DIR=<dir> claude /login` to clipboard, "Test" button calls the probe and shows `OK (820ms)` or `✗ <stderr_tail>` for 6s.
- `tests/test_routes_accounts_health.py` (new) — 12 cases covering shape extension, sentinel-based auth_failure detection, idle/active/untested derivation, and probe endpoint paths (success / failure / missing CLI / secret-resolution failure).

## Key decisions

- **Reuse the auth-cooldown sentinel as the auth_failure_recent signal** — the file already exists from the Phase 6 monitor and its mtime is the canonical "we recently saw OAuth break" timestamp. Avoids a `task_runs.auth_failure` migration and keeps the data flow single-source.
- **Health is derived in the route, not stored** — fields drift fast (sentinel comes and goes; idle threshold is policy not data) so computing on read is cheaper than maintaining a denormalized column.
- **3 SQL queries per account** for runs_24h / failures_24h / cost_30d. O(3N) for the list view is fine at this user base size; resist the urge to GROUP BY until the dashboard hits 50+ accounts.
- **Probe endpoint always returns HTTP 200**. FE branches on `ok:false` instead of catch+parse — fewer paths, simpler UI.
- **Probe uses `--max-turns 1`** so a successful auth check costs at most one tiny round-trip on the operator's quota.
- **"Copy re-login" is clipboard-only** — no subprocess from the browser; the operator runs the command in their terminal. Same ToS-compliant approach as T1's add-account flow.

## Gotchas

- The probe inherits `os.environ.copy()` and only overrides `CLAUDE_CONFIG_DIR` / `ANTHROPIC_API_KEY` — never mutates `os.environ` in place.
- Secret resolution errors return `ok:false` with a sanitized message — the resolved secret value never appears in the response.
- The `auth_failure_recent` boolean and the cooldown sentinel use the same constants from `core/notify.py`. If the cooldown window changes there, this endpoint follows automatically.
