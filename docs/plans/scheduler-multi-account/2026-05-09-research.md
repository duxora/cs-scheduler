# Research: Multi-account Claude scheduler — auth & isolation

**Ticket:** #1214 (Phase 1 of epic #1213)
**Date:** 2026-05-09
**Decision:** **CLAUDE_CONFIG_DIR profiles** for subscription-backed accounts; **API key (Console)** as a fallback for accounts that need it. **No OAuth.**

---

## 1. Question

Can the `claude-scheduler` use a documented OAuth/PKCE flow to run scheduled `claude -p` invocations under multiple Anthropic accounts (Pro / Max / personal / work)?

## 2. Finding

**No.** As of 2026-05-09, Anthropic does not expose a documented OAuth flow for third-party automation against claude.ai or Claude Code. The reverse is now actively enforced:

- **2026-01-09** — Anthropic shipped a server-side change that blocks OAuth tokens issued by Free/Pro/Max plans from working anywhere except claude.ai and the official `claude` CLI itself.
- **2026-02-19** — Documentation updated to clarify that using subscription OAuth tokens with third-party tools (including the Agent SDK) violates the Consumer Terms of Service.
- **2026-04-04 15:00 ET** — Hard enforcement: Pro/Max/Team subscriptions no longer cover usage through third-party harnesses (OpenClaw, OpenCode, NanoClaw, etc.).

The supported authentication paths today are:
- **OAuth (Pro/Max/Team subscription)** — reserved for first-party `claude` CLI and claude.ai. Routing it through any third-party process is a ToS violation.
- **API key (Console)** — required for any custom tooling, SDK integration, or Agent SDK use. Pay per token; no plan caps.

Because `claude-scheduler` shells out to the **first-party `claude` CLI**, it is on the supported path *as long as the CLI itself is the one consuming the credentials*. The scheduler never owns or forwards tokens — it just sets `CLAUDE_CONFIG_DIR` and lets the CLI authenticate normally.

## 3. Options compared

| Option | Multi-account? | Cost model | ToS-safe | Operational cost |
|---|---|---|---|---|
| **A. CLAUDE_CONFIG_DIR per account** | ✅ — one profile dir per account, swap via env var at exec time | Subscription quota (Pro/Max) for that account | ✅ — uses first-party CLI exactly as Anthropic intends | Low: `mkdir`, `claude /login` once per account |
| **B. ANTHROPIC_API_KEY per account** | ✅ — keys stored in Keychain, looked up at exec time | Pay per token, Console billing | ✅ — explicitly the supported path for tooling | Medium: per-account billing/budgets to track |
| **C. OAuth/PKCE flow built ourselves** | ❌ technically possible to scrape, but | Subscription quota | ❌ violates ToS as of 2026-02-19 | Drop |
| **D. Mix A + B** | ✅ — use subscription where available, key as fallback | Hybrid | ✅ | Low+ |

**Decision: D. Hybrid.** Default to A (CLAUDE_CONFIG_DIR) for personal Pro/Max accounts. Use B (API key) for accounts that need workspace separation (work) or where a subscription isn't available. Reject C entirely.

## 4. CLAUDE_CONFIG_DIR contract

The `claude` CLI honors the `CLAUDE_CONFIG_DIR` env var: when set, the CLI reads/writes its config (auth tokens, MCP registrations, recent session state) under that directory instead of the default `~/.claude/`.

Implication for the scheduler:

```bash
# One-time setup per account
mkdir -p ~/.claude-profiles/personal-pro
CLAUDE_CONFIG_DIR=~/.claude-profiles/personal-pro claude /login
# → opens browser, completes OAuth, writes credentials inside that dir

mkdir -p ~/.claude-profiles/work-max
CLAUDE_CONFIG_DIR=~/.claude-profiles/work-max claude /login
```

```bash
# At dispatch time
CLAUDE_CONFIG_DIR=~/.claude-profiles/personal-pro \
  claude -p "..." --model opus --max-turns 10 --output-format json
```

The scheduler never sees the OAuth token — it's stored inside the per-account config dir, owned and refreshed by the first-party CLI. The DB only stores the **path** to that dir.

## 5. API key contract (fallback)

```bash
# At dispatch time, when account.kind = "api_key"
ANTHROPIC_API_KEY=$(security find-generic-password -s claude-scheduler-<account_name> -w) \
  claude -p "..." --model opus
```

Keys live in **macOS Keychain** (`security` CLI) or 1Password CLI (`op read op://Vault/Item/credential`). The DB stores only an opaque reference, e.g. `keychain:claude-scheduler-work` or `op://Personal/claude-api-key/credential`. **Never the key value.**

## 6. claude_accounts table — derived shape

Confirms the schema sketched in the epic:

```sql
CREATE TABLE claude_accounts (
  id            TEXT PRIMARY KEY,        -- UUID
  name          TEXT NOT NULL UNIQUE,    -- 'personal-pro', 'work-max', 'api-fallback'
  kind          TEXT NOT NULL CHECK (kind IN ('config_dir', 'api_key')),
  config_dir    TEXT,                    -- abs path; required when kind='config_dir'
  api_key_ref   TEXT,                    -- 'keychain:<name>' or 'op://...' ; required when kind='api_key'
  plan_tier     TEXT,                    -- 'pro' | 'max' | 'team' | 'api' | NULL
  is_default    INTEGER DEFAULT 0,
  created_at    TEXT NOT NULL,
  last_used_at  TEXT
);
CREATE UNIQUE INDEX claude_accounts_default ON claude_accounts(is_default) WHERE is_default = 1;
```

CHECK constraints: exactly one of `config_dir` / `api_key_ref` populated based on `kind`.

## 7. Phase 6 disposition

Phase 6 (OAuth migration) was gated on this research. **Mark it deferred** — revisit only if Anthropic publishes a third-party auth surface for subscription accounts. Until then, the gap is closed by Phase 2–4 (CLAUDE_CONFIG_DIR + API-key hybrid).

## 8. Risks / non-obvious

1. **Profile dir refresh** — OAuth tokens stored in a profile dir expire (typically ~30 days). The first-party CLI refreshes them automatically as long as the CLI runs interactively *occasionally*. If a profile sits idle past expiry, the next scheduled run fails with an auth error. **Mitigation:** monitor `task_runs` for auth-failure stderr; emit a Telegram alert prompting `CLAUDE_CONFIG_DIR=<dir> claude /login`. Track in Phase 4.
2. **Profile dir size** — each profile carries MCP server configs and a session cache. Don't symlink configs across profiles; let each profile own its own copy. ~50 MB per profile is normal.
3. **Concurrent runs on same profile** — the CLI's credential file is read+written. Two simultaneous `claude` runs on the *same* profile dir can race. The existing per-task lockfile in `executor.py` is keyed on task slug, not on profile — Phase 4 must add a *profile-level* lock to prevent two different tasks bound to the same account from racing the auth refresh.
4. **API-key billing surprise** — API keys are usage-billed. Phase 5 (advisor/brainstorm with Opus) can rack up cost fast. Reuse the existing `budget_usd` field on `Task` and enforce per-task; consider adding a per-account daily ceiling in a follow-up.

## 10. Re-research check (2026-05-09, post-Phase-5)

Re-ran the OAuth-surface check after Phases 2–5 shipped. **Verdict: still deferred** — no signal of reversal between 2026-04-04 (the ban) and 2026-05-09 (today). Findings:

- **Anthropic changelog (May 2026)** — only auth-related entries are bug fixes (Mantle `x-api-key` header, OAuth 401 retry loop when `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1`, MCP OAuth menu). No new third-party auth surface. No BYOS API. No service-principal flow.
- **Pricing/plans page** — still draws the hard line: subscriptions cover claude.ai + first-party Claude Code only; programmatic = `ANTHROPIC_API_KEY`, billed per-token.
- **`claude setup-token` / `sk-ant-oat01-*` route** — still rejected by the API as of 2026-02-20; not re-enabled.
- **Open canonical request**: `anthropics/claude-code` issue **#18340** (API-compatible key from `claude setup-token`, or subscription-linked key in console). Open, not accepted, no maintainer commitment — this is the trigger to watch.
- **Workarounds** (browser-automation, token extraction, custom replay) all violate the 2026-04-04 ToS update. Do not ship.

**Re-open #1219 only if one of:**
1. `anthropics/claude-code` #18340 is marked accepted or shipped.
2. `console.anthropic.com` exposes a subscription-linked API key.
3. `claude setup-token` tokens start being honored by the API again.

Until then, Phase 1's hybrid (config_dir for subscriptions + `ANTHROPIC_API_KEY` for API tier) remains the only ToS-compliant path.
